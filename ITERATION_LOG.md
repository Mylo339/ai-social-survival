# AI Social Survival Iteration Log

日期：2026-06-05

## 本轮目标

把原型从“英语练习器”升级成更有趣的“社交语气实验室”。

## 参考到的现有产品功能

- Duolingo Max：Roleplay / Video Call，说明 AI 语言学习的核心卖点是角色化实时对话，而不只是题库。
- Loora：自然对话、实时反馈、结束后的关键改进点。
- Praktika：AI avatar、真实场景、发音和流畅度反馈、进度追踪。
- Speak / 类似产品：真实生活场景、即时反馈、语音练习。

## 本轮加入的功能

### 1. 语气版本

新增四种语气：

- 礼貌版：适合第一次见面、服务场景和面试；
- 自然版：日常聊天，不端着；
- 自信版：清楚表达，不卑不亢；
- 不耐烦版：用来观察社交风险，不建议现实乱用。

用户可以在同一场景中随时切换语气，快捷回复会跟着变化。

### 2. 角色化对话

每个场景现在都有具体 NPC：

- Sam：咖啡店店员；
- Jess：设计课同学；
- Morgan：兼职面试经理。

每个角色有自己的身份、情绪和场景压力。

### 3. Mission / Risk

每个场景增加：

- Mission：用户要完成的社交任务；
- Risk：这个场景最容易出问题的地方。

这让产品更像游戏，而不是普通课程。

### 4. Vibe check

对话中会根据用户语气给即时反馈：

- 太直接；
- 太短；
- 礼貌缓冲不错；
- 自信表达不错；
- 不耐烦语气高危。

### 5. 结果页升级

结果页新增：

- 语气相关徽章；
- 本场最佳复盘；
- 根据语气历史调整尴尬值；
- 不耐烦版会增加社交风险。

## 当前仍未做

- 真实 AI 对话；
- 真实发音评分；
- 账号与历史记录；
- 场景解锁；
- 分享卡片；
- 小红书传播素材；
- 移动端专门适配；
- 真实用户测试。

## 下一轮建议

下一轮应该做“剧情感”和“可传播性”：

1. 增加社交 HP 条和冷场警报；
2. 做一张可分享的结果卡；
3. 增加一个“室友说 Can you keep it down?” 场景；
4. 加入“本地人到底什么意思”解释模块；
5. 为每个场景增加 3 个结局：顺利、普通、社死边缘。

# 第二轮自主迭代

日期：2026-06-05

## 本轮目标

让产品更像“社交生存游戏”，而不只是“语气选择聊天框”。

## 新增功能

### 1. Social HP

每个场景增加社交血量。礼貌、自然、自信表达会保持或提升 HP；不耐烦、太短、太硬的表达会扣 HP。

HP 会实时影响冷场提示：

- 空气正常；
- 冷场预警；
- 空气开始结冰；
- 社交 HP 见底。

### 2. 冷场警报

每次用户回复后，系统会根据语气和文本即时更新气氛状态。这让用户能看到“语法没错但关系变冷”的差别。

### 3. 隐藏含义解释

每个场景加入“他们真实想表达什么”：

- 咖啡店：店员只是想快速确认选项；
- 小组作业：同学想知道你是否愿意参与；
- 面试：经理真正关心可靠性；
- 室友：对方想解决噪音，但还想保持和平。

### 4. 新场景：室友说 Can you keep it down?

这是更贴近留学生生活的场景，训练用户处理轻微冲突：

- 道歉；
- 解释；
- 保持边界；
- 不把合租关系推向冷战。

### 5. 场景结局

每个场景现在有三种结局：

- 好结局；
- 普通结局；
- 高尴尬/冷场结局。

结局由尴尬值和 Social HP 共同决定。

### 6. 分享文案

结果页生成可分享文案，包含：

- 场景；
- 结局；
- 自然度；
- 尴尬值；
- Social HP。

## 测试

新增本地 smoke test：

```text
tests/smoke-test.mjs
```

测试覆盖：

- 四个场景是否渲染；
- 语气选择是否渲染；
- 室友场景是否能启动；
- 不耐烦语气是否扣 Social HP；
- 是否生成冷场提示；
- 是否生成坏结局；
- 是否生成徽章；
- 是否生成分享文案。

运行结果：

```text
Smoke test passed: roommate flow, tone switching, Social HP, ending, badges, and share line.
```

## 当前限制

当前 in-app browser 对本地 `file://` 页面自动化读取有限制，因此本轮使用本地模拟 DOM smoke test 进行核心流程验证。视觉检查仍需要用户刷新页面后查看。

## 下一轮建议

1. 增加“结果分享卡”视觉样式，而不仅是文字；
2. 增加“教授邮件”和“租房看房”场景；
3. 让每个 NPC 对不同语气产生不同后续回复；
4. 增加每日挑战和 streak；
5. 接入真实 AI，让用户自由回答而不是只靠预设脚本。

# 第三轮自主迭代

日期：2026-06-05

## 本轮目标

提升 UI 趣味性、交互强度、回复多样性和语音输入存在感，并补充更多自动测试。

## 新增功能

### 1. XP 与 Tone Combo

对话页新增 XP 和 Tone Combo。礼貌、自然、自信且较完整的表达会增加 XP；连续保持良好语气会形成 Combo。

### 2. Live Reaction

对话页新增实时反应条。用户每次回复后，系统会给出更即时的社交反馈，例如：

- 礼貌缓冲到位；
- 语法可能没错，但气氛掉了一截；
- 清楚、有主见；
- 能用，但像把对话轻轻关上了。

### 3. 多样化回复

每个语气不再只显示固定两条快捷回复。现在系统会扩展出更多同类表达，并提供“换一批说法”按钮，让用户比较同一个意思的不同社交手感。

### 4. NPC 回复多样化

NPC 不再机械重复固定下一句。系统会根据用户语气和 HP 变化添加不同回应前缀：

- 礼貌回应更柔和；
- 自信回应更认可；
- 不耐烦回应会出现轻微疏离；
- HP 扣得多时，NPC 会显得更谨慎。

### 5. 语音模式强化

页面新增 Voice mode 面板，明确说明当前使用真实浏览器麦克风输入。语音识别结果会先进入输入框，用户可修改后再发送。

### 6. 本地 HTTP 测试服务器

新增：

```text
local-server.mjs
```

可用来通过 `http://127.0.0.1:4177` 测试页面资源，而不是只依赖 `file://`。

## 测试

新增：

```text
tests/interaction-test.mjs
```

覆盖：

- 模拟真实语音输入；
- 四个场景；
- 四种语气；
- 换一批回复；
- XP；
- 结局；
- 分享文案复制。

本轮测试结果：

```text

smoke-test.mjs: passed
interaction-test.mjs: passed
HTTP resource check: index.html/app.js/styles.css all returned 200
```

## 浏览器测试限制

尝试通过 in-app browser 打开 `file://` 和 `localhost/127.0.0.1` 时，当前环境分别受到安全策略和浏览器端拦截限制。因此本轮采用：

- 本地 HTTP 资源检查；
- 模拟 DOM smoke test；
- 更完整的 interaction test。

这能验证核心交互逻辑，但最终视觉仍需要人工刷新页面查看。

## 下一轮建议

1. 做真正的可视化分享卡；

# 语音输入修复

日期：2026-06-05

## 问题

用户反馈语音输入在当前页面不可用。原因不是按钮逻辑本身，而是当前 Codex in-app browser 打开的 `file://` 页面无法稳定使用浏览器 Web Speech API 或麦克风权限。

## 修复

### 1. 增加语音诊断

页面现在会显示：

- 当前浏览器是否支持 Web Speech API；
- 当前是否是 `file://` 页面；
- 如果麦克风权限被拒绝，给出具体提示；
- 如果没有识别到声音，提示检查音量和权限；
- 如果语音服务被浏览器阻止，提示换 Chrome/Edge 打开本地服务。

### 2. 增加测试麦克风按钮

新增“测试麦克风”按钮，与 Speak 使用同一套真实语音识别逻辑。识别结果会先进入输入框，不会自动发送。

### 3. 增加 Chrome/Edge 启动脚本

新增：

```text
open-in-chrome.ps1
```

该脚本会：

1. 启动本地服务器；
2. 打开 `http://127.0.0.1:4177`；
3. 使用 Chrome 或 Edge，这两个浏览器更可能支持 Web Speech API。

### 4. 修复启动脚本的本地服务

首次脚本使用 Adobe Node 时，本地环境出现 `Path/PATH` 冲突；随后改为优先使用 Codex 自带 Node，但后台 Node 服务在当前环境中仍不稳定。

之后改为 PowerShell TCP 静态服务器：

```text
static-server.ps1
```

该服务器不依赖 Node、Python 或 `HttpListener`，只用 `System.Net.Sockets.TcpListener` 提供静态文件。

## 验证

运行：

```text
powershell.exe -NoProfile -ExecutionPolicy Bypass -File E:\CodexProjects\ai-social-survival\open-in-chrome.ps1
```

随后资源检查：

```text
http://127.0.0.1:4177/index.html -> 200
http://127.0.0.1:4177/app.js -> 200
http://127.0.0.1:4177/styles.css -> 200
```

自动测试仍通过：

```text
app.js syntax check: passed
smoke-test.mjs: passed
interaction-test.mjs: passed
```

# 社交事故应急台视觉重构

日期：2026-06-06

## 为什么重做

上一版虽然整洁，但大量使用深色玻璃卡片、圆角、胶囊按钮和均匀网格，整体容易被看成普通 AI SaaS 或后台面板，缺少“社交生存游戏”的记忆点。

## 新视觉方向

将产品定义成一套“社交事故应急台 / 生存现场手册”：

- 暖纸张底色、粗黑线框、安全红与酸性绿；
- 事件编号、警戒条、现场印章和通讯终端；
- 不对称排版、轻微旋转、硬边阴影和档案纸张感；
- 场景卡变成不同颜色的事件文件；
- 对话气泡带通讯编号和明显角色方向；
- 关系温度会把侧栏从稳定绿切换为警告黄或危险红；
- 不同结局会改变复盘报告的状态色；
- 结果页改成 After Action Report，而不是普通数据卡片。

## 保留的可用性原则

- 颜色不是唯一状态提示，仍保留文字说明；
- 键盘焦点清晰；
- 支持减少动画偏好；
- 移动端取消卡片旋转并改为单列；
- 功能逻辑、隐私边界和透明评分说明保持不变。

## 验证

- 原有 smoke、interaction、rubric 测试继续通过；
- 新增 UI 测试，检查统一视觉身份、关系温度状态、结局状态与减少动画支持。

# Web Beta 降噪与正式发布准备

日期：2026-06-06

## 为什么重做

“社交事故应急台”虽然有辨识度，但高饱和色、粗框、硬阴影、旋转和跑马灯同时出现，长时间使用会造成视觉负担。Web Beta 改为更安静的“真实生活沟通指南”。

## 核心变化

- 视觉收敛为海军蓝主色、珊瑚色强调和柔和状态色；
- 移除持续动画、旋转卡片、酸性绿色和大量硬阴影；
- 增加留白、统一圆角与轻量边框，简化首页和练习页信息层级；
- 场景从 4 个扩展到 8 个，并按日常、学业、工作、居住分类；
- 新增退换商品、导师答疑、临时换班和看房费用场景；
- 关键词判断改为完整单词或短语匹配，避免把 `planet` 中的 `plan` 误判为相关回答；
- 增加 16 次完整旅程测试、发布资源测试和服务器接口测试；
- 增加 Docker Render Blueprint、GitHub 持续测试和部署说明。

## 当前验证

- 8 个场景、32 轮规则和全部建议语气通过校验；
- 练习与挑战模式共 16 次完整旅程通过；
- 语音模拟、筛选、历史、收藏、分享和离线反馈通过；
- 静态资源、安全响应头、反馈接口、非法事件拒绝和 AI 回退通过；
- 当前桌面内置浏览器被本地地址安全策略阻止，仍需在正式 HTTPS 地址和真实手机上进行最终麦克风验证。
